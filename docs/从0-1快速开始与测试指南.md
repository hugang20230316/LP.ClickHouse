# LP.ClickHouse 从0到1快速开始与测试指南

> 这份指南的目标不是“把命令列出来”，而是让你在一台新的开发机上，按顺序执行后，**一次完成环境启动、建表、造数、查询验证和单元测试**。  
> 如果你严格按下面顺序执行，最终应看到：ClickHouse 可访问、API 可访问、`/api/schema/init` 返回 200、造数成功、分析接口有数据、单元测试通过。

## 先说清楚：这份指南会做什么

你会得到 3 类可验证结果：

1. **基础设施可用**：Docker 中的 ClickHouse、Prometheus、Grafana 启动成功。
2. **示例 API 可用**：本地 `http://localhost:5066` 能响应，且能成功初始化表结构。
3. **功能验证通过**：写入演示数据后，分析接口能查询到结果，单元测试也能通过。

## 前置要求

### 系统环境
- **操作系统**：Windows 10/11、macOS 或 Linux
- **.NET SDK**：.NET 8.0 或更高版本
- **Docker Desktop / Docker Engine**：用于运行 ClickHouse 和监控栈
- **终端**：推荐 PowerShell 7、Windows Terminal、bash 或 zsh

### 环境检查
先确认基础命令可用：

```bash
# 检查 .NET SDK
 dotnet --version

# 检查 Docker
 docker --version
 docker compose version
```

**预期结果**：
- `dotnet --version` 输出 `8.0.x` 或更高
- `docker compose version` 输出 `2.x` 或更高

## 步骤1：进入项目目录

```bash
cd e:\Projects\LP\LP.ClickHouse
```

后续命令默认都在**项目根目录**执行，这样最不容易因为当前目录错误而失败。

## 步骤2：启动 ClickHouse 和监控栈

这里用 `-f docker/docker-compose.yml` 明确指定 Compose 文件，避免你终端里如果配置了别的 `COMPOSE_FILE` 或当前目录不对时，误操作到其他项目。

```bash
docker compose -f docker/docker-compose.yml -p lp-clickhouse up -d
```

### 验证容器状态

```bash
docker compose -f docker/docker-compose.yml -p lp-clickhouse ps
```

**预期结果**：至少能看到下面 3 个服务：
- `clickhouse`
- `prometheus`
- `grafana`

### 验证 ClickHouse 已真正可访问

这一步非常关键。  
**只有这一步成功后，后面的 `schema/init` 才一定能成功。**

```bash
curl -u default:clickhouse "http://localhost:8123/?query=SELECT%201"
```

**预期结果**：返回

```text
1
```

如果这里不返回 `1`，不要继续执行后面的 API 初始化步骤，先看文末“故障排除”。

## 步骤3：构建项目

```bash
dotnet build LP.ClickHouse.slnx -v minimal
```

**预期结果**：
- `0 个错误`
- `已成功生成`

## 步骤4：启动示例 API

```bash
dotnet run --project samples/LP.ClickHouse.Api
```

**预期结果**：终端出现类似日志：

```text
Now listening on: http://localhost:5066
```

### 验证 API 已启动

打开另一个终端窗口，在项目根目录执行：

```bash
curl "http://localhost:5066/"
```

**预期结果**：返回包含下面字段的 JSON：
- `message`
- `utcNow`

## 步骤5：检查 ClickHouse 连通性

先做轻量检查，再做建表。这样如果数据库没起来，能更早发现问题。

```bash
curl "http://localhost:5066/api/schema/ping"
```

**预期结果**：

```json
{"success":true}
```

如果这里返回 `false` 或 `503`，说明 API 已启动，但它还连不上 ClickHouse；这时回到步骤2重新检查 `8123`。

## 步骤6：初始化数据库表结构

```bash
curl -X POST "http://localhost:5066/api/schema/init"
```

**预期结果**：返回成功消息，例如：

```json
{"message":"ClickHouse 数据库和示例表已初始化"}
```

## 步骤7：写入演示数据

为了兼容 Windows PowerShell 和 bash，这里统一使用**单行命令**，避免多行续行符差异导致复制后失败。

```bash
curl -X POST "http://localhost:5066/api/demodata/seed" -H "Content-Type: application/json" -d '{"logCount":5000,"eventCount":10000}'
```

**预期结果**：返回类似结构：
- `insertedLogRows`
- `insertedEventRows`
- `generatedAtUtc`

说明：`event` 实际写入条数通常会**大于你传入的 `10000`**，因为示例数据按“浏览 → 点击 → 完成”的漏斗链路生成，一次基础事件可能展开成多条记录，这是正常现象。

## 步骤8：功能验证（API 测试）

这一步的目标不是把所有接口都打一遍，而是确认“日志分析”和“行为分析”两类能力都能返回真实结果。

### 8.1 查询最近错误日志

```bash
curl "http://localhost:5066/api/analytics/recent-errors?hours=24&limit=5"
```

**预期结果**：返回 JSON 数组；通常能看到 `statusCode` 为 5xx 的记录。

### 8.2 查询 API 统计

```bash
curl "http://localhost:5066/api/analytics/api-stats?hours=24&limit=10"
```

**预期结果**：返回 JSON 数组，字段通常包括：
- `bucketStart`
- `apiPath`
- `requestCount`
- `errorCount`
- `avgResponseTimeMs`
- `p95ResponseTimeMs`

### 8.3 查询 DAU（日活）

```bash
curl "http://localhost:5066/api/analytics/dau?days=7"
```

**预期结果**：返回 JSON 数组，字段包括：
- `activityDate`
- `activeUsers`

### 8.4 查询漏斗分析

```bash
curl "http://localhost:5066/api/analytics/funnel?days=7"
```

**预期结果**：返回 JSON 对象，字段包括：
- `viewedCount`
- `clickedCount`
- `completedCount`
- `viewToClickRate`
- `clickToCompletionRate`

## 步骤9：运行单元测试

```bash
dotnet test LP.ClickHouse.slnx -v minimal
```

如果你只想先验证连接字符串相关测试，也可以执行：

```bash
dotnet test tests/LP.ClickHouse.Tests --filter "FullyQualifiedName~ConnectionStringBuilderTests"
```

**预期结果**：测试通过，没有失败。

## 步骤10：查看 ClickHouse 中的真实数据（可选）

如果你想确认数据确实已经写入 ClickHouse，而不只是 API 返回了成功，可以直接进容器查看。

```bash
docker exec -it lp-clickhouse clickhouse-client --user default --password clickhouse
```

进入后执行：

```sql
SHOW TABLES;
SELECT count() FROM lp_observability.api_logs;
SELECT count() FROM lp_observability.user_events;
```

**预期结果**：
- 能看到 `api_logs` 和 `user_events`
- 两张表都有大于 0 的记录数

退出客户端：

```sql
exit
```

## 步骤11：停止和清理（可选）

```bash
docker compose -f docker/docker-compose.yml -p lp-clickhouse down
```

如果你要连数据卷一起清理：

```bash
docker compose -f docker/docker-compose.yml -p lp-clickhouse down -v
```

## 一次跑通时，关键检查点有哪些

如果你想确认自己是不是“按正确节奏在推进”，看这 6 个点就够了：

1. `curl -u default:clickhouse "http://localhost:8123/?query=SELECT%201"` 返回 `1`
2. `dotnet build LP.ClickHouse.slnx -v minimal` 成功
3. `http://localhost:5066/` 能返回 JSON
4. `http://localhost:5066/api/schema/ping` 返回 `{"success":true}`
5. `POST /api/schema/init` 返回 200
6. `POST /api/demodata/seed` 后，4 个分析接口都能查到结果

## 故障排除

### 1. `schema/init` 报无法连接 `localhost:8123`

这不是建表 SQL 有问题，而是 **ClickHouse 根本还没监听 8123**。  
按下面顺序排查：

```bash
docker compose -f docker/docker-compose.yml -p lp-clickhouse ps
curl -u default:clickhouse "http://localhost:8123/?query=SELECT%201"
```

只有 `curl` 返回 `1`，你再去调用：

```bash
curl -X POST "http://localhost:5066/api/schema/init"
```

### 2. `docker compose ps` 看到的不是这个项目的容器

请确认你用的是下面这条命令，而不是只写 `docker compose ps`：

```bash
docker compose -f docker/docker-compose.yml -p lp-clickhouse ps
```

这样可以强制指定当前项目的 Compose 文件和项目名，避免误操作其他 compose 项目。

### 3. API 已启动，但 `ping` 返回 `false`

说明 ASP.NET Core 进程正常，但 ClickHouse 还没准备好。  
通常原因有：
- ClickHouse 镜像还在启动中
- 8123 端口被占用
- Docker 容器未成功起来

先执行：

```bash
curl -u default:clickhouse "http://localhost:8123/?query=SELECT%201"
```

返回 `1` 后再重试。

### 4. PowerShell 下复制多行 `curl` 失败

本指南已经尽量使用**单行命令**，就是为了避免：
- bash 使用 `\` 续行
- PowerShell 不识别同样写法

如果你在 Windows 上，优先直接复制本指南里的**单行命令**。
